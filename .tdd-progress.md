## Feature Description

Implement network discovery (`Zenoh.scout`) and session information queries (`Session.zid`, `Session.routersZid`, `Session.peersZid`). This adds 6 new C shim functions, 3 new Dart classes (`ZenohId`, `WhatAmI`, `Hello`), 3 new Session methods, 1 new Zenoh static method, and 2 CLI examples (`z_scout.dart`, `z_info.dart`). These features are standalone -- no counterpart entity is needed to test the basic functionality (a session always has its own ZID).

**Reference spec:** `docs/design/phase-05-scout-info-revised.md`
**Cross-cutting patterns:** `docs/design/cross-cutting-patterns.md`

## Implementation Notes

### Architectural Decisions

1. **ZenohId.toHexString via native call:** Uses `zd_id_to_string` (which calls `z_id_to_string`) for consistent LSB-first hex formatting matching zenoh-c/zenoh-cpp output.

2. **Synchronous collection for info ZIDs:** `z_info_routers_zid` and `z_info_peers_zid` use closures that fire synchronously during the C call. The C shim collects ZIDs into a caller-provided buffer (16 bytes per ZID, `max_count` entries). Avoids NativePort overhead.

3. **NativePort for scout:** Scout blocks for `timeout_ms` and the number of Hello results is unbounded. The C shim posts each Hello as a Dart_CObject array to the NativePort, followed by a null sentinel.

4. **CLI examples in `example/`:** Design doc references `packages/zenoh/bin/` but project convention places CLI examples in `packages/zenoh/example/`.

5. **Config consumption in scout:** `z_scout` consumes the config via `z_config_move`. Dart must call `config.markConsumed()` after.

6. **Two-session testing for peersZid:** Use unique TCP port 17460 for this test group (following the project convention of unique ports per test group to avoid conflicts).

### Deferred Features (per design doc)
- Whatami bitmask combinations (e.g., `Z_WHAT_ROUTER_PEER = 3`) -- the `what` parameter accepts raw int but Dart API exposes simple enum values only
- `z_hello_clone`/`z_hello_drop` -- Hello data is extracted in C callback, Dart receives pure data
- Scout `what` filter as typed enum parameter -- exposed as raw `int?` for now
